CoLabGame.

A benchmark and framework for comparing single-, hierarchical-, and decentralized multi-agent LLM systems on real-world computer-use tasks.

role · sole author advisor · u. potsdam year · 2025 status · defended
// tldr Centralized hierarchical agent architectures consistently beat both flat multi-agent councils and single-agent baselines on robustness, efficiency, and cost — across GPT-5, GPT-5-mini, Gemini-2.5, and Claude-4-Sonnet, on real OSWorld desktop tasks.

Problem

Most LLM-agent benchmarks test single agents on toy tasks. Real desktop work — clicking through GUIs, dropping into a terminal, juggling files and forms — needs coordination across heterogeneous interfaces. There weren't good evaluations for it, and the few multi-agent papers I read disagreed about whether coordination overhead was worth the price.

What I built

CoLabGame is a Python evaluation framework that runs the same OSWorld task suite against three classes of agent system:

  • Single agent — one model, one prompt, one trajectory.
  • Decentralized multi-agent — peer agents that communicate to coordinate, no central planner.
  • Hierarchical multi-agent — a planner agent decomposes the task and delegates to skill-specialized executors.

Each class is parameterized by role-based collaboration topologies, orchestration protocols, and the underlying LLM. Failure modes, coordination cost, and operational $/token are all logged.

Hierarchical setups recovered from individual-agent failures more often, completed multi-step tasks more reliably, and ran cheaper than decentralized peer councils — which spent a surprising fraction of their budget on inter-agent chatter that didn't change the plan.

What I'd do differently

  • Test on a wider task distribution — OSWorld leans desktop-heavy; would love to add browser-only and terminal-only baselines.
  • Vary the planner model and the executor models independently. There's a cheap-planner / strong-executor configuration I never got to try.
  • Add a learned router that picks topology per-task instead of fixing it upfront.
